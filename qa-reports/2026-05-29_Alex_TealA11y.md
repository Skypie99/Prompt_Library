| Pair | Hex | Background | Ratio | Required | Result |
|------|-----|------------|-------|----------|--------|
| white label on teal.500 chip | `#FFFFFF` | `#2F9E96` | **3.25:1** | ≥4.5 normal text, ≥3.0 UI/large | FAIL — see Issue 1 |

**Context:** `CategoryChips` and `TagChips` both use `bg-teal-500 text-white` for active state. The chip button text is `text-sm font-medium` (14px, weight 500). WCAG AA normal text requires 4.5:1. WCAG large text (≥18pt or ≥14pt bold = weight ≥700) requires 3.0:1. `font-medium` (500) does not meet WCAG's bold threshold, so 4.5:1 applies. At 3.25:1 this **fails AA for normal text**.

Dani's spec correctly identifies 3.25:1 as passing the 3:1 UI component threshold (WCAG SC 1.4.11), but the text inside the chip is the failing element — it is subject to SC 1.4.3 (text contrast), not SC 1.4.11 (non-text). The chip border/bg shape passes as a UI component; the white label text does not pass as text.

## Color-Not-Only Check

### ARIA state

Both `CategoryChip` and the tag buttons in `TagChips` use `aria-pressed={isActive}`. This correctly conveys active/inactive state to screen readers independent of color. PromptCard star button uses `aria-pressed={isFavorite}`.

- `aria-pressed` on filter chips: PRESENT — conveyed without color reliance

### Visual differentiator beyond color

Active chips (`border-teal-500 bg-teal-500 text-white`) vs inactive chips (`border-border bg-surface text-ink-muted`):
- Color change: YES (teal fill vs neutral border)
- Border weight: SAME (both use `border`, 1px)
- Icon/checkmark: NONE
- Shadow: Active chips add `shadow-sm` — a subtle elevation change

The visual distinction relies primarily on color (fill vs unfilled). There is no border weight change, no icon, no checkmark. For sighted users with color vision differences, this could be ambiguous. However, the `aria-pressed` provides the screen reader path. For low-vision sighted users (color weak), the filled-vs-unfilled contrast of 3.25:1 for the shape itself (teal.500 vs white background) is the signal — and 3.25:1 meets the UI component threshold.

**Assessment:** Marginally acceptable for WCAG 2.1 (the shape distinguishes), but a added border-weight change or icon on active would be a stronger non-color signal. This is POLISH, not BLOCKER.

## Issues

### BLOCKER 1 — Active chip text: white on teal.500 fails AA normal text (3.25:1 vs required 4.5:1)

**WCAG:** SC 1.4.3 (Contrast — Minimum), Level AA
**Where:** `CategoryChips.tsx` and `TagChips.tsx` — active state `text-white` on `bg-teal-500`
**Ratio:** 3.25:1 (requirement: 4.5:1 for 14px/font-medium)

**Fix options (any one):**
1. Darken active chip background to teal.600 (`#238178`): white on teal.600 = 4.68:1 — PASS. Trade-off: slightly heavier look.
2. Use dark teal text instead of white: teal.900 (`#103B37`) on teal.200 background (6.xx:1) — completely different visual but passes.
3. Use `font-semibold` or `font-bold` (weight ≥700): if text is bold ≥14px, 3:1 suffices. teal.500 at 3.25:1 would pass AA large text. This is the lightest visual change. **Recommended.**

Option 3 (add `font-bold` or `font-semibold`) is the minimal change — just add `font-semibold` to the active variant in both components. Visually nearly identical but crosses the WCAG bold threshold.

---

### BLOCKER 2 — Focus rings: teal.400 fails SC 2.4.11 against cream (2.30:1 vs required 3.0:1)

**WCAG:** SC 2.4.11 (Focus Appearance), Level AA (new in WCAG 2.2)
**Where:** All components using `focus-visible:ring-teal-400` with `ring-offset-cream` (CategoryChips, TagChips, Header, SettingsModal save button, PromptDetail tags, most interactive elements)
**PromptCard:** even worse — uses `ring-teal-300` (1.77:1)
**Ratio:** teal.400 = 2.30:1 vs cream; teal.300 = 1.77:1 vs cream (both fail)

**Fix:**
Replace all `focus-visible:ring-teal-400` with `focus-visible:ring-teal-500`. teal.500 achieves 3.02:1 vs cream — just passes.
Replace `focus-visible:ring-teal-300` (PromptCard) with `focus-visible:ring-teal-500`.

This is a global find-and-replace across ~15 files. Both `teal-400` and `teal-300` ring usages should be bumped to `teal-500`.

**Note:** SettingsModal close button already uses `ring-teal-500` — that's the correct pattern. The save button in the same modal uses `ring-teal-400` — inconsistency to fix.

---

### POLISH 3 — teal.600 hover text on cream: 4.35:1 (just below 4.5 threshold)

**WCAG:** SC 1.4.3, Level AA
**Where:** `TagChips.tsx` "show more" / "show fewer" buttons; `CategoryChips.tsx` inactive chip hover state when rendered on cream background (page bg). The inactive chip is `bg-surface` (`#FFFDF9`) and `hover:text-teal-600` achieves 4.61:1 on surface — PASSES. The failure only occurs where the element appears on the page's cream background and the `bg-surface` is visually absent.

**Severity:** POLISH — this is a hover state (not resting), and hover text is not guaranteed to be visually read. However, the "show more" button on TagChips has no other bg and renders directly on cream — its hover text at 4.35:1 is technically a violation.

**Fix option:** Use `teal.700` (`#1C6660`) for hover text on cream surfaces instead of `teal.600`. teal.700 on cream = 6.46:1. Trade-off: slightly darker hover state.

---

### POLISH 4 — Seafoam category stripe: 2.88:1 against card surface (just below 3.0)

**WCAG:** SC 1.4.11 (Non-text Contrast), Level AA
**Where:** `categoryColor.ts` — slot 0 (seafoam, `#6FA09A`) as the 3px left border stripe
**Ratio:** 2.88:1 vs card surface `#FFFDF9`; 2.93:1 vs white

Dani's spec noted this sits "at a glance" in the ~3:1 tier, but the actual computed value (2.88:1) falls just short of the 3.0 threshold against the card surface.

**Fix option:** Darken seafoam slightly. `#678D87` achieves approximately 3.05:1 vs surface while remaining visually in the same family. Alternatively, accept the 0.12-point gap as a practical near-miss given the 3px width (very few pixels fail). This is the least severe issue but technically a violation.

## Spec Accuracy Check
