## F3a — Overloaded Retry

**Lines 943–953 in PromptDetail.tsx**

### Findings

**PASS — proper `<button>` element.** The retry button uses `<button onClick={handleRetry}>`, not a `<div>` or `<span>` with an onClick. Keyboard-reachable by default (tab order flows naturally through the error block).

**PASS — accessible name present.** `aria-label="Retry"` is set on the button (line 948). It has visible text "Retry" as well, so the label is not lying about the visible content.

**PASS — semantically distinct from rate-limit retry.** The rate-limit retry (lines 920–936) lives in a separate conditional branch (`error.kind === "rate-limit"`). The overloaded retry is in its own branch (`error.kind === "overloaded"`, lines 943–953). The two are mutually exclusive — only one error kind can be active at a time, so they cannot both render simultaneously. No semantic conflict.

**POLISH — `aria-label="Retry"` is minimal but sufficient.** The button reads "Retry" to screen readers; context is provided by the error message above (`error.message` in the same block). A richer label like `aria-label="Retry — server overloaded"` would remove the dependency on reading the surrounding paragraph first, but this is not a blocker.

**POLISH — both rate-limit and overloaded use the same `handleRetry()` function.** This is intentional and correct per Shamus's report; the shared path is fine. However, if a future engineer audits only `handleRetry` in isolation, the function name doesn't distinguish the two paths. No a11y impact, just a minor code-readability note for Gary.

## F3c — Variable Warning

**Lines 769–812 in PromptDetail.tsx**

### Findings

**PASS — `role="alert"` is present** on the warning container `<div>` (line 771). This causes screen readers to announce the warning immediately when it mounts without requiring focus. For a warning that appears in response to a user action (clicking Run), live-region announcement on mount is exactly correct — the user just clicked something and needs to know why the run didn't start.

**PASS — warning message is count-aware.** The text correctly adapts:
- 1 missing: "1 variable is empty — run anyway?"
- N missing: "N variables are empty — run anyway?"

The message gives a number (actionable: the user knows how many to fill) and a clear binary choice ("run anyway?" implies the action word exists in the buttons). For a screen reader user who only hears the alert text, the message communicates the situation.

**POLISH — "Fill it" label is ambiguous for screen reader users.** The button reads "Fill it" (line 797). A screen reader user navigating by buttons hears only "Fill it" — they cannot see the amber warning box providing context. "Fill empty variables" or "Go to first empty field" would be self-contained. This is a POLISH (not a BLOCKER) because the `role="alert"` fires first and provides the context sentence before the user tabs to the button. The sequence (alert announces, user tabs, user hears "Fill it") is workable but not ideal.

**PASS — "Run anyway" label is unambiguous.** No disambiguation needed; "Run anyway" is self-contained.

**ISSUE (POLISH) — Focus management after warning dismisses.** 
- When "Fill it" is clicked: focus is correctly moved to the first empty field (line 789–793, `panelRef.current?.querySelector(...).focus()`). Good.
- When "Run anyway" is clicked: warning dismisses and `runWithValues(values)` is called. Focus is not explicitly placed anywhere. It stays on the "Run anyway" button, which is then removed from the DOM. This likely causes focus to drop to `<body>` — the "focus loss on DOM removal" anti-pattern. For screen reader users this means the next Tab from an unknown starting point, which is confusing mid-workflow.
  - **Suggested fix:** In the "Run anyway" `onClick`, before calling `runWithValues`, move focus to the response panel container (`responsePanelRef.current?.focus()`) or back to the "Run with Claude" button. The response panel would need `tabIndex={-1}` to receive programmatic focus.

**PASS — ⌘↵ bypass is discoverable.** The hint `⌘↵ to run` appears in the status line just below the action buttons (line 826) via `<kbd>` element. It is visible at all times when the form is open. Keyboard users who read this hint know the shortcut exists. The shortcut bypass in `handleModalKeyDown` (line 446–450) calls `handleRun()` — but note: when `showUnfilledWarning` is already visible, `⌘↵` calls `handleRun()` again, which re-fires `setShowUnfilledWarning(true)` (since variables are still unfilled). It does not actually bypass the warning for a second ⌘↵ press. The spec says "⌘↵ bypasses the warning"; in practice ⌘↵ only bypasses the warning if pressed before it appears. This is a **BLOCKER** — see Issues section below.

## F3d — Expand/Collapse Toggle

**Lines 866–880 in PromptDetail.tsx**

### Findings

**PASS — `aria-label` reflects current state.** The button uses:
```
aria-label={responseExpanded ? "Collapse response" : "Expand response"}
```
(line 869). The label is dynamic and always accurate. "Expand response" / "Collapse response" names both the action and the object — this is the right pattern.

**PASS — proper `<button>` element with `type="button"`.** Keyboard-reachable, correct semantics.

**PASS — visual label matches aria-label.** The button text toggles between "Expand" and "Collapse" (lines 879–880), consistent with the `aria-label`. The visual label is slightly shorter ("Expand" vs. "Expand response") but the `aria-label` carries the full text for screen readers. Acceptable.

**ISSUE (POLISH) — `aria-expanded` attribute is missing.** The button communicates state only via its `aria-label` text ("Expand response" vs. "Collapse response"). The correct pattern for a toggle that shows/hides content is `aria-expanded={responseExpanded}` on the button element. Without it, screen reader users who navigate by buttons and look at the button role/state (rather than its label) get no state signal. The `aria-label` text change alone is not machine-readable state — some SR + AT combinations surface state separately.
  - **Suggested fix:** Add `aria-expanded={responseExpanded}` to the button at line 866.

**ISSUE (POLISH) — `aria-controls` is absent.** The button expands/collapses the response `<div>` below it, but there is no `aria-controls="some-id"` linking them. `aria-controls` tells a screen reader "this button controls that region", allowing the user to jump directly to the controlled content. Without it, the relationship is only visual.
  - **Suggested fix:** Add `id="response-content"` to the response content `<div>` (line 957) and `aria-controls="response-content"` to the toggle button.

## Issues

### BLOCKER

**B1 — ⌘↵ does not bypass the unfilled warning when the warning is already visible.**

- Location: `handleModalKeyDown` (line 446), `handleRun` (lines 395–401)
- Problem: `⌘↵` calls `handleRun()`. `handleRun()` checks `filledCount < variables.length` and, if true, sets `showUnfilledWarning(true)`. If the warning is already showing, pressing `⌘↵` does nothing useful — it just re-sets the same state. The spec (Quinn F3 spec + Shamus report) explicitly states ⌘↵ is the "power-user path" to bypass the warning. The intent is: once the warning is visible, ⌘↵ should call `runWithValues(values)` directly.
- Suggested fix:
  ```typescript
  function handleRun() {
    if (variables.length > 0 && filledCount < variables.length) {
      setShowUnfilledWarning(true);
      return;
    }
    void runWithValues(values);
  }

  function handleModalKeyDown(event: React.KeyboardEvent) {
    if ((event.metaKey || event.ctrlKey) && event.key === "Enter") {
      event.preventDefault();
      if (!running) {
        if (showUnfilledWarning) {
          // ⌘↵ bypasses the warning — run directly
          setShowUnfilledWarning(false);
          void runWithValues(values);
        } else {
          handleRun();
        }
      }
      return;
    }
    // ... rest unchanged
  }
  ```

### POLISH

**P1 — Focus drops to `<body>` when "Run anyway" is clicked.**
- Location: lines 803–807
- "Run anyway" removes the warning from the DOM without explicitly placing focus. Focus likely lands on `<body>`. 
- Fix: before calling `runWithValues`, move focus to a stable element — the response panel (add `tabIndex={-1}`) or back to the Run button via a `ref`.

**P2 — "Fill it" button label is context-dependent.**
- Location: line 797
- Suggested: `aria-label="Fill empty variables"` on the "Fill it" button, while keeping visible text as "Fill it".

**P3 — F3d toggle missing `aria-expanded`.**
- Location: line 866
- Add `aria-expanded={responseExpanded}` to the Expand/Collapse button.

**P4 — F3d toggle missing `aria-controls`.**
- Location: line 866 and the response content div (line 957)
- Add matching `id` + `aria-controls` to associate button with controlled region.

**P5 — F3a overloaded Retry label is minimal.**
- Location: line 948
- Consider `aria-label="Retry — server is overloaded"` to be context-independent. Low priority.

## ESCALATIONS

**B1 (⌘↵ bypass)** should block merge. The spec explicitly calls ⌘↵ a bypass path and Shamus's report confirms it was built that way — but the implementation does not deliver it when the warning is already showing. This is a behavior gap, not a style preference. Shamus to fix on the feature branch before Rory merges.

P1 (Run anyway focus loss) is a POLISH but affects keyboard users meaningfully — recommend fixing alongside B1 since both are in the same warning block.

P3 + P4 (aria-expanded / aria-controls on F3d) are standard WCAG 4.1.2 pattern requirements for expand/collapse widgets. Recommend fixing; they are one-liners.

P2 and P5 are low-priority polish — can go in a follow-up cycle.
